# 04 Packet Serialization

## Зачем это

Этот урок нужен, чтобы руками почувствовать:
- что пакет по сети — это просто буфер байтов;
- что у каждого поля есть точное смещение;
- почему `header + payload` лучше, чем слепой каст `struct` в память;
- как явно писать и читать числа из буфера.

## Артефакт урока

Одна маленькая программа, которая:
- собирает `POSITION`-подобный пакет в `uint8_t buffer[10]`;
- печатает его в hex;
- потом разбирает тот же буфер обратно в поля.

Ожидаемый вывод:

```text
hex: 50 4f 53 00 03 01 2c 02 bc 07
tag: POS
type: 3
x: 300
y: 700
dimension: 7
```

## Формат пакета в этом уроке

```text
[0..3]  tag        4 bytes   "POS\0"
[4]     type       1 byte
[5..6]  x          2 bytes   uint16 big-endian
[7..8]  y          2 bytes   uint16 big-endian
[9]     dimension  1 byte
```

Итого: `10` байт.

## Почему здесь big-endian

Чтобы не зависеть от архитектуры машины и сразу привязать упражнение к сетевому мышлению:
- старший байт идёт первым;
- младший байт — вторым.

Это тот же класс задач, который в реальном коде часто решается через `htons()` / `ntohs()`.

## Что сделать

1. Открой `template.c`.
2. Реализуй:
   - `write_u16_be()`
   - `read_u16_be()`
   - сборку пакета в `build_position_packet()`
   - разбор пакета в `parse_position_packet()`
3. Добейся точного вывода из секции “Артефакт урока”.

## Что здесь важно понять

- буфер пакета — это массив байтов, а не “волшебная структура”;
- смещение каждого поля надо держать явно;
- одинаковый размер констант не делает их взаимозаменяемыми;
- сериализация и парсинг должны соглашаться о формате.

## Критерий прохождения

Урок можно считать пройденным, если ты можешь объяснить:
- почему `x` и `y` занимают именно эти смещения;
- почему сериализация и парсинг должны использовать один и тот же порядок байт;
- почему опасно надеяться, что layout `struct` случайно совпадёт с сетевым форматом;
- чем этот ручной буфер отличается от простого `memcpy(struct, ...)`.
