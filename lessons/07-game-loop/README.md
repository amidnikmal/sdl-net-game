# 07 Game Loop

## Зачем это

Это финальный и уже более верхнеуровневый урок.

Здесь цель не “ещё раз вызвать SDL-функцию”, а понять саму форму кадра:

```text
read input/network -> update state -> send changes -> render -> wait next frame
```

В этом репозитории именно такой порядок и держит игру живой.

## Артефакт урока

Маленький симулятор из 3 кадров, который печатает trace игрового цикла.

Ожидаемый вывод:

```text
frame 1: net=none input=right update=x=1 send=yes render=x=1
frame 2: net=left input=none update=x=0 send=no render=x=0
frame 3: net=none input=up update=x=0 send=no render=x=0
```

## Что здесь моделируется

- `net`
  - событие, пришедшее из сети;
- `input`
  - локальное действие игрока;
- `update`
  - новое состояние после применения событий;
- `send`
  - надо ли отправлять обновление другим;
- `render`
  - что реально увидит экран.

## Что сделать

1. Открой `template.c`.
2. Реализуй:
   - применение сетевого события;
   - применение локального ввода;
   - решение “надо ли отправлять пакет”;
   - печать trace для каждого кадра.

## Правила симуляции

- Начальное состояние: `x = 0`.
- Кадр 1:
  - сеть ничего не прислала
  - локальный ввод: `right`
- Кадр 2:
  - сеть прислала `left`
  - локальный ввод: `none`
- Кадр 3:
  - сеть ничего не прислала
  - локальный ввод: `up`

Смысл команд:
- `right` -> `x += 1`
- `left` -> `x -= 1`
- `up` не меняет `x`, только показывает, что не любой input влияет на сетевое состояние

Правило `send=yes/no`:
- `yes`, если локальный игрок изменил `x` в этом кадре;
- `no`, если `x` изменился только из-за сети или не изменился вообще.

## Что здесь важно понять

- игровой цикл — это порядок стадий, а не просто бесконечный `while`;
- чтение сети и локальный input могут влиять на один и тот же state;
- не каждое изменение состояния надо отправлять обратно по сети;
- render должен опираться на уже обновлённое состояние кадра.

## Почему это верхний слой

Потому что это уже ближе к:
- архитектуре кадра;
- причинности изменений;
- сетевой синхронизации;
- проектированию state flow.

То есть это ближе к design/review мышлению, чем к “как вызвать одну функцию”.

## Критерий прохождения

Урок можно считать пройденным, если ты можешь объяснить:
- почему `read/update/send/render` — это разные стадии;
- почему изменение от сети не надо слать обратно как своё локальное действие;
- почему `render` должен идти после `update`;
- как этот порядок связан с `client.c` в основном проекте.
