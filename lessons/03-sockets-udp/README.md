# 03 Sockets UDP

## Зачем это

Этот урок нужен, чтобы руками пройти самый маленький рабочий UDP-обмен:

- создать сокет;
- привязать серверный сокет через `bind()`;
- отправить датаграмму через `sendto()`;
- принять её через `recvfrom()`;
- увидеть адресную структуру `sockaddr_in`;
- использовать `htons()` для порта.

## Артефакт урока

Одна маленькая программа на C, которая внутри одного процесса делает roundtrip:

1. клиентский UDP-сокет отправляет `ping`;
2. серверный UDP-сокет принимает `ping`;
3. сервер отправляет `pong` обратно;
4. клиент принимает `pong`.

Ожидаемый вывод:

```text
server received: ping
client received: pong
```

## Почему один процесс, а не два

Потому что здесь цель не “настроить настоящий многопроцессный сервер”, а почувствовать минимальную механику:
- `socket`
- `bind`
- `sendto`
- `recvfrom`

Это маленький учебный стенд, а не production-модель.

## Что сделать

1. Открой `template.c`.
2. Заполни `TODO`, чтобы:
   - создать 2 UDP-сокета;
   - настроить `server_addr`;
   - привязать сервер к `127.0.0.1:9091`;
   - отправить `ping`;
   - принять `ping` на сервере;
   - отправить `pong` обратно;
   - принять `pong` на клиенте.

## Что здесь важно понять

- `socket(AF_INET, SOCK_DGRAM, 0)` = UDP-сокет IPv4
- `bind()` нужен серверной стороне, чтобы занять конкретный адрес и порт
- `sendto()` отправляет в конкретный адрес
- `recvfrom()` получает данные и может вернуть адрес отправителя
- `htons()` переводит порт в network byte order

## Критерий прохождения

Урок можно считать пройденным, если ты можешь объяснить:
- зачем серверу нужен `bind()`;
- чем `sendto()` отличается от `recvfrom()`;
- почему порт проходит через `htons()`;
- почему в этой программе клиент может получить ответ, даже без явного `bind()` на своей стороне.

## Минимальный смысл

- UDP не создаёт поток байтов как TCP: он посылает отдельные датаграммы.
- Сервер знает, кому отвечать, потому что `recvfrom()` дал адрес отправителя.
- Этот урок не про “надёжность сети”, а про базовую механику адреса и обмена.
