# 06 Client Server Protocol

## Зачем это

Это уже более верхний слой.

Здесь цель не “ещё раз написать сокет”, а понять:
- какие бывают типы пакетов;
- как сервер хранит состояние игроков;
- что сервер делает при `CONNECT`;
- что сервер делает при `POSITION`;
- кому какой пакет надо переслать.

То есть урок про **протокол и state machine**, а не про низкоуровневую механику сети.

## Артефакт урока

Одна маленькая программа, которая симулирует серверную логику для 3 событий:

1. `CONNECT alice`
2. `CONNECT bob`
3. `POSITION alice`

Ожидаемый вывод:

```text
connect alice: sync=0 broadcast=1 total=1
connect bob: sync=1 broadcast=2 total=2
position alice: forward=1
```

Смысл:
- `sync` = сколько уже подключённых игроков надо отправить новому игроку;
- `broadcast` = скольким игрокам сервер рассылает `CONNECT` после подключения;
- `forward` = скольким игрокам сервер надо переслать `POSITION`, кроме автора.

## Что сделать

1. Открой `template.c`.
2. Заполни `TODO`, чтобы:
   - находить игрока по `id`;
   - добавлять нового игрока в состояние;
   - считать `sync` и `broadcast` для `CONNECT`;
   - считать `forward` для `POSITION`.

## Что здесь важно понять

- протокол — это не только формат байтов, но и **правила поведения**;
- `CONNECT` и `POSITION` делают разные вещи;
- сервер хранит состояние (`кто подключён`);
- пересылка зависит не только от пакета, но и от текущего состояния.

## Почему это верхний слой

Потому что здесь уже не так важно, через `recvfrom()` или не через него пришёл пакет.

Главный вопрос:
- что **логически** должен сделать сервер, когда получил событие?

Это ближе к архитектуре, ревью и проектированию протокола, чем к “просто написать код”.

## Критерий прохождения

Урок можно считать пройденным, если ты можешь объяснить:
- почему `CONNECT` требует и sync, и broadcast;
- почему `POSITION` не надо слать обратно автору;
- почему протокол без серверного состояния неполон;
- как этот слой связан с реальными багами в `serv_network.c`.
