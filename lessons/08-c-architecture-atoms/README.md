# 08 C Architecture Atoms

## Что это

Это не ещё один “большой курс по C”.

Это набор маленьких архитектурных атомов:
- каждый атом = одна best practice;
- каждый атом = маленький игровой стенд;
- каждый атом = один короткий вывод, который можно проверить.

Идея:
- не учить “архитектуру” как абстрактную философию;
- а трогать руками маленькие контракты, границы и правила сборки системы.

## Для чего это в контексте `sdl-net-game`

Основной курс `01..07` даёт:
- компиляцию
- make
- сокеты
- сериализацию
- SDL
- протокол
- game loop

Этот трек даёт слой выше:
- ownership
- API boundaries
- error handling
- single source of truth
- domain vs transport
- context вместо глобалов

Это как раз те места, где текущий проект чаще всего ломается не по синтаксису, а по архитектурной дисциплине.

## Порядок атомов

1. `01-ownership-contract`
2. `02-opaque-handle`
3. `03-error-return`
4. `04-single-source-of-truth`
5. `05-domain-vs-transport`
6. `06-no-globals-context`

## Как проходить

- Не надо проходить всё за раз.
- Один атом = 15-30 минут.
- Смысл не в “закрыть все папки”, а в том, чтобы начать узнавать эти паттерны в живом коде.

Хороший цикл:
- открыть `template`
- добить до точного вывода
- сравнить с `solution`
- потом найти аналогичную проблему в `src/`

## Что именно считать успехом

Не:
- “я выучил архитектуру”

А:
- “я теперь узнаю ownership bug”
- “я теперь понимаю, зачем opaque type”
- “я теперь не пихаю exit() в helper”
- “я теперь вижу, где у меня два источника правды”

## Связь с реальными проблемами проекта

- `01-ownership-contract`
  - связан с утечками в `build_*_packet()` и call-site ownership
- `02-opaque-handle`
  - связан с плохими границами между клиентом, сервером и SDL
- `03-error-return`
  - связан с `exit()` внутри сетевых helper-путей
- `04-single-source-of-truth`
  - связан с рассинхроном `x_coord` и `rect.x`
- `05-domain-vs-transport`
  - связан с `network.h` / `network.c` и ручной сериализацией
- `06-no-globals-context`
  - связан с глобальной `player` и размытым ownership состояния

## Стоп-критерий

Если ты прошёл хотя бы первые `3` атома и начал узнавать эти паттерны в коде проекта, этого уже достаточно для пользы.

Не надо превращать и этот трек в бесконечную систему.
